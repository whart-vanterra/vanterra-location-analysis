# Location Intelligence v3.0 — Design Spec

## Purpose

Identify the highest-ROI cities for Vanterra brands to open physical offices and dominate local SEO, GEO, and Google Map Pack results for home services (foundation repair, basement waterproofing, crawl space repair, concrete lifting).

The core ROI equation:

> **Demand** (people searching) × **Convertibility** (homeowners who can pay) × **Opportunity** (low competition) × **Portfolio Fit** (no cannibalization) = **Best city to open next**

## Problem Statement

The current system (v2.6) has two core issues:

1. **Scoring model produces wrong rankings.** Log normalization destroys the search volume signal — Philadelphia (780/mo) and Levittown (60/mo) end up only 4.2 points apart despite 13x more demand. Owner-occupied % dominates everything. CRM data is zero for most expansion cities, making 20% of scoring dead weight.

2. **Data pipeline is fragile.** Eight Python scripts live in `/tmp` with no organization, no shared config, no clear run order. Data files are scattered between `/tmp`, `data/`, and Supabase with no single source of truth.

## Approach

**Python pipeline + Next.js static export (Hybrid).**

- Python handles data fetching and scoring — it works, just needs organization.
- Next.js handles presentation — component-based UI makes interactive features (sliders, popovers, sparklines) clean to build and maintain.
- Static export to Vercel for deployment — no server costs, no runtime dependencies.
- Supabase as canonical data store — pipeline writes there, export pulls from there.

This is a strategic tool, not a daily-use app. Built once, generated, shared with stakeholders.

### Relationship to v3 Rebuild Plan

This design spec supersedes `docs/v3-rebuild-plan.md` in the `vanterra-location-analysis` repo. The scoring model is identical, but the architecture is upgraded from loose `/tmp` scripts + monolithic HTML to an organized Python pipeline + Next.js static export with Supabase as the canonical data store. Key differences:

- Population backfill (Phase 0 in rebuild plan) is handled by `load_existing.py`
- City splitting (Phase 2 in rebuild plan) is deferred to Phase 4 here — get base scoring right first
- Report is Next.js instead of generated HTML

## Supabase Schema

All tables prefixed `loc_` to identify project ownership.

### loc_cities (Canonical City Reference)

All other tables reference cities via `city_key` — a normalized identifier derived from `lower(city) || '|' || lower(state)` (e.g., `philadelphia|pa`, `st. louis|mo`). This table is the canonical source of truth for city identity, preventing join mismatches from inconsistent naming across data sources (Census "Virginia Beach city" vs Google Ads "Virginia Beach").

| Column | Type | Description |
|--------|------|-------------|
| city_key | text PK | Normalized `lower(city)\|lower(state)` |
| city | text | Display name (canonical) |
| state | text | State abbreviation |
| lat | float | City centroid |
| lng | float | City centroid |
| population | int | Canonical population (SUM from ZIP data) |
| created_at | timestamptz | |

**Normalization rules** (applied by all pipeline scripts before lookup/insert):
1. Trim whitespace
2. Lowercase for key generation
3. Strip " city", " town", " CDP" suffixes
4. Normalize "Saint"/"St.", "Fort"/"Ft.", "Mount"/"Mt." to canonical form
5. Display name preserves original casing

## Scoring Engine

### Design Principle

Two implementations sharing one config:

- **Python (`pipeline/score.py`)** — canonical scoring for official pipeline runs
- **TypeScript (`app/src/lib/scoring.ts`)** — client-side mirror for live browser experimentation

Both read `config.json` for all weights, thresholds, and constants. No magic numbers in code.

**Dual-engine verification:** `export.py` includes the canonical Python-computed scores in the exported JSON. The TypeScript engine has a verification mode (runs on build) that compares its output against the canonical scores for all cities. Tolerance: ±0.1 points. Build fails if engines diverge. The `config_hash` in `loc_scored_recommendations` is also embedded in the exported JSON so the client can detect stale data.

### Scoring Input Assembly

Before scoring, `score.py` assembles a flat input record per brand × city by joining:

```sql
SELECT
  bcm.brand_id,
  c.city_key, c.city, c.state, c.lat, c.lng, c.population,
  sv.foundation_vol, sv.basement_vol, sv.crawlspace_vol, sv.concrete_vol,
  sv.avg_competition_index,
  cd.owner_occupied_pct, cd.median_household_income, cd.median_year_built,
  crm.leads, crm.jobs, crm.revenue,
  b.keyword_weights, b.existing_locations, b.confidence_tier
FROM loc_brand_city_mappings bcm
JOIN loc_cities c ON c.city_key = bcm.city_key
JOIN loc_brands b ON b.brand_id = bcm.brand_id
LEFT JOIN loc_search_volumes sv ON sv.city_key = c.city_key
LEFT JOIN loc_census_demographics cd ON cd.city_key = c.city_key
LEFT JOIN loc_crm_data crm ON crm.brand_id = bcm.brand_id AND crm.city_key = c.city_key
```

**Missing data handling:**
- If a city has no search volume data: skip it (search volume is the primary signal — no data = no recommendation)
- If a city has no census data: use state median values from available cities in that state. If no state data, use national median. Flag `data_confidence` as LOW.
- If a city has no CRM data: normal case for expansion cities — CRM badge = null, leads/jobs/revenue = 0
- `lat`, `lng`, `population` come from `loc_cities` (canonical, not per-brand)

### Scoring Algorithm

For each brand × city:

#### 1. Market Demand (0–45 pts)

**Search Volume** (70% = 31.5 pts effective):
```
brand_vol = foundation_vol * kw.foundation + basement_vol * kw.basement + ...
max_vol = max(brand_vol) across all cities for this brand
raw = sqrt(brand_vol / max_vol)
search_score = raw * 31.5
```
Per-brand normalization prevents low-volume-state brands from scoring near zero.

**Population** (30% = 13.5 pts effective):
```
capped_pop = min(population, 500000)
max_pop = min(max(population) for brand, 500000)
raw = sqrt(capped_pop / max_pop)
pop_score = raw * 13.5
```

#### 2. Market Quality (0–30 pts)

**Owner-Occupied %** (55% = 16.5 pts effective):
```
raw = clamp((owner_pct - 0.30) / (0.90 - 0.30), 0, 1)
owner_score = raw * 16.5
```

**Median Household Income** (30% = 9 pts effective):
```
raw = clamp((income - 40000) / (120000 - 40000), 0, 1)
income_score = raw * 9
```

**Median Year Built** (15% = 4.5 pts effective):
```
raw = clamp((config.year_built_baseline - year_built) / (config.year_built_baseline - config.year_built_oldest), 0, 1)
year_score = raw * 4.5
```

#### 3. Competitive Opportunity (0–25 pts)

**Competition Index — Inverted** (50% = 12.5 pts effective):
```
raw = 1 - (comp_index / 100)
comp_score = raw * 12.5
```

**Same-Brand Distance** (30% = 7.5 pts effective):
```
if dist < 15mi:  raw = 0
elif dist < 30mi: raw = (dist - 15) / 15
elif dist < 60mi: raw = 1.0
else: raw = 0.8  (isolation penalty)
same_brand_score = raw * 7.5
```

**Sister Brand Overlap** (20% = 5 pts effective):

Count Vanterra sister brands with an existing office within `sister_brand_radius_mi` (config default: 50mi).

```
if sisters == 0: raw = 1.0
elif sisters <= 2: raw = 0.7
else: raw = 0.4
sister_score = raw * 5
```

#### 4. Composite Score
```
base = demand + quality + competition  (0-100)
```

#### 5. Portfolio Gap (Toggle Overlay, 0–25 pts)

Cross-brand distance bonus:
```
threshold = density_adjusted_threshold(density)
if dist < threshold * 0.5: gap = 0
elif dist < threshold: gap = 25 * (dist - threshold*0.5) / (threshold*0.5)
else: gap = 25

display = (base + gap) / 125 * 100  (when enabled)
```

#### 6. CRM Badge (Not in score)

| CRM Jobs | Badge |
|----------|-------|
| 10+ | PROVEN |
| 1–9 | SIGNAL |
| 0 | *(none)* |

#### 7. Sensitivity Flag

For each of the 3 top-level weights, increase it by 5 points and decrease the other two proportionally to maintain sum = 100. Rescore all cities for the brand. If any city's rank changes by 5+ positions in any perturbation, flag it as sensitive. This produces 6 perturbations per brand (3 weights × up/down).

### Validation Criteria

The model passes if:
1. Cities with 500+ monthly searches always rank top 10 for their brand (unless competition is extreme)
2. Philadelphia scores above Levittown for every brand covering both
3. Score spread: top > 75, median ~50, bottom < 35
4. Portfolio gap toggle visibly reorders rankings
5. CRM-proven cities get badge but aren't artificially boosted above better expansion opportunities

## Pipeline Resilience

**Idempotent operations:** All pipeline scripts use upserts (ON CONFLICT UPDATE) so re-runs are safe. `score.py` truncates `loc_scored_recommendations` before inserting (scores are always fully recomputed).

**API fetch scripts** (`fetch_search_volume.py`, `fetch_census.py`):
- Rate limiting: 2-second delay between API calls
- Retry: 3 attempts with exponential backoff on 429/5xx responses
- Checkpoint/resume: save progress every 20 cities. On restart, skip cities already in Supabase.
- Partial failure: log failures, continue with remaining cities, report summary at end

**`app/src/data/` is generated output** — gitignored. The canonical config is `pipeline/config.json` only. The exported `config.json` in `app/src/data/` is a snapshot produced by `export.py`.
